# ikiw — LLM Wiki ナレッジベース

LLM ベースの個人ナレッジベース管理 skill。フレームワーク、データベース、ベクトル検索に一切依存せず、純粋に prompt 駆動で動作し、ファイルの読み書きができる LLM agent であれば何でも利用可能。

Karpathy の LLM Wiki パターンに着想を得ている。

## コマンド

```
ikiw init <path>                    ナレッジベースを初期化、対話で SCHEMA.md を生成
ikiw summary                        一括要約生成（未処理の記事を検出）
ikiw summary <filename>             指定した記事の要約を生成
ikiw query "質問"                    ナレッジベースを検索
ikiw wiki "テーマ"                   wiki ページを生成
ikiw wiki "テーマ" --style <style>   wiki を生成しライティングスタイルを適用
ikiw wiki "テーマ" --design <design> wiki を生成しビジュアルスタイルで HTML 出力
ikiw write "テーマ" --style <style>  ナレッジベースの内容に基づきスタイルライティング
ikiw ingest                         新規記事を処理（要約生成、wiki 更新チェック）
ikiw setup-summary                  要約アシスタント、対話で要約 prompt を定義
```

自然言語トリガーにも対応：
- 「ナレッジベースを検索」「調べて」「関連記事はある？」
- 「要約を生成して」「新しい記事を処理して」
- 「wiki ページを作って」「このテーマをまとめて」
- 「ナレッジベースの XX に関する内容」

## コアコンセプト

**ナレッジベース = 1つのディレクトリに3つの要素：**

```
my-wiki/
├── raw/              # 元記事、変更しない
├── summaries.md      # 要約インデックス、システムの中核
├── wiki/             # クエリ結果のキャッシュ、必要に応じて生成
└── SCHEMA.md         # このナレッジベースの設定
```

- `raw/` は元素材。読み取り専用、絶対に変更しない
- `summaries.md` は LLM のグローバルマップ。各記事につき1件の要約
- `wiki/` は必要に応じて生成されるナレッジページ。本質はクエリキャッシュ
- `SCHEMA.md` はこのナレッジベースのルールとカスタム prompt を定義する

## 機能

### 1. 要約生成

raw/ 内の記事を読み、SCHEMA.md で定義された要約 prompt に従って要約を生成し、summaries.md に追記する。

**単一記事の処理：**
1. 記事の内容を読み取る
2. 要約 prompt に従って要約を生成する
3. summaries.md の末尾に追記する

**一括処理：**
1. raw/ 内の全ファイルを一覧する
2. summaries.md に既存のファイル名と比較し、未処理の記事を特定する
3. 1件ずつ要約を生成して追記する
4. 最も効率的な処理方法を自律的に判断する（逐次処理、スクリプトによる並行処理、バッチ処理など）

### 2. クエリ

ユーザーが具体的な意図を持って質問し、ナレッジベースから回答を見つける。

**フロー：**
1. summaries.md を読み取る（全量要約インデックス）
2. 質問に基づき、要約からどの記事が関連するかを判断する
3. 関連する原文を読み取る（通常 10〜30 件）
4. 総合的に回答し、具体的な記事を引用する
5. 回答に長期的な再利用価値がある場合、wiki ページとして保存することを提案する

### 3. Wiki 生成

特定テーマの知識を構造化されたページにまとめる。

**フロー：**
1. summaries.md を読み取り、関連記事をフィルタリングする
2. 関連する原文を読み取る
3. SCHEMA.md で定義された wiki prompt に従ってページを生成する
4. フレームワーク型構成：各知識ポイント = 一文の要約 + 原文へのリンク
5. wiki/ ディレクトリに保存する

**Wiki はキャッシュであり、権威ある情報源ではない：**
- 期限切れになり得るし、再構築も可能
- 新しい記事が追加されたら、関連する wiki の更新が必要になる場合がある
- 完璧さではなく、実用性を追求する

### 4. スタイル出力（オプション）

デフォルト出力は markdown。より良いビジュアル表現が必要な場合、design-md スタイルと組み合わせて styled HTML を生成できる。

**使い方：**
- ユーザーがスタイルを指定する：「Notion 風で出力して」「Claude 風でウェブページを生成して」
- `designs/` ディレクトリ下の対応するスタイルファイルを読み取る
- スタイルルールに従って wiki コンテンツを styled HTML ページにレンダリングする
- HTML ファイルをナレッジベースの `wiki/` ディレクトリに保存する（.md と同名、拡張子は .html）

**組み込みスタイル：** 本 skill の `designs/` ディレクトリに配置
- `notion` — 温かみのあるミニマル、セリフ見出し、柔らかなサーフェス
- `claude` — テラコッタトーン、エディトリアルレイアウト、書籍的な雰囲気
- `linear.app` — 超ミニマル、精密、パープルアクセント
- `stripe` — 象徴的なパープルグラデーション、軽やかでエレガント
- `vercel` — 白黒の精密さ、Geist フォント
- `mintlify` — グリーンアクセント、読みやすさ重視

**スタイルの追加：** https://github.com/VoltAgent/awesome-design-md
- インストール：`npx getdesign@latest add <style-name>`
- 生成された DESIGN.md をリネームして `designs/` ディレクトリに配置する

### 5. スタイルライティング

ナレッジベースの内容に基づき、指定されたライティングスタイルテンプレートで記事を出力する。

**フロー：**
1. コンテンツソースを取得する（wiki ページ、クエリ結果、または指定の原文）
2. `styles/` ディレクトリ下の対応するライティングスタイルテンプレートを読み取る
3. スタイルテンプレートに従ってコンテンツをリライトする
4. markdown で出力。design-md と組み合わせて styled HTML の生成も可能

**組み込みライティングスタイル：** 本 skill の `styles/` ディレクトリに配置
- ユーザーは独自のスタイルテンプレートを追加可能。`styles/` ディレクトリに配置するだけでよい

**3層のコンポーザブル構成：**
- コンテンツ層（wiki prompt）→ 何を書くかを決定
- スタイル層（styles/）→ どう書くかを決定
- ビジュアル層（designs/）→ どう見せるかを決定

ユーザーが「小紅書の販売促進 wiki を公式アカウント風で記事にして、Notion スタイルでウェブページを生成して」と言えば、agent は wiki コンテンツ、ライティングスタイルテンプレート、ビジュアルスタイルを順に読み取り、層ごとにレンダリングして出力する。

### 6. 新規記事の登録

新しい記事が raw/ に追加された後の処理フロー：
1. raw/ 内で summaries.md に未登録の記事を検出する
2. 要約を生成し、summaries.md に追記する
3. wiki/ 内に更新が必要な関連ページがないか確認し、ユーザーに通知する

## 新規ナレッジベースの初期化

ユーザーが「ナレッジベースを作って」と言った場合：

1. ナレッジベースのディレクトリパスを確認する
2. ディレクトリ構造を作成する：raw/、wiki/
3. 対話を通じてユーザーの記事タイプと関心事を把握する
4. 対話に基づきカスタムの SCHEMA.md を生成する（要約 prompt、wiki prompt）
5. 空の summaries.md を作成する
6. ユーザーに記事を raw/ に配置するよう案内する
7. 要約の生成を開始する

## 要約アシスタント

初回利用時、対話を通じてユーザーに適した要約 prompt を定義する：

- 「記事はどんなタイプですか？」（技術ブログ、ビジネスケース、学術論文…）
- 「検索時に最も気になる情報は？」（方法論、データ、人物、タイムライン…）
- 「要約の長さはどのくらいが適切ですか？」（1〜2文の超簡潔、3〜5文の標準、1段落の詳細）

回答に基づき要約 prompt を生成し、SCHEMA.md に書き込む。ユーザーはいつでも調整可能。

## マルチナレッジベース対応

1つの skill で複数のナレッジベースを管理でき、それぞれが独自の SCHEMA.md を持つ：

```
~/wikis/生財有術/SCHEMA.md
~/wikis/交易策略/SCHEMA.md
~/wikis/技術笔記/SCHEMA.md
```

クエリ時にナレッジベースを指定するか、agent が質問内容に応じてどのナレッジベースを検索すべきか自動的に判断する。

## スケールの境界

- **1万件以内：** summaries.md を一括読み込み可能、追加のインフラ不要
- **1万件超：** ベクトル検索を導入して粗いフィルタリングが必要。要約データはそのまま embedding に活用可能
- 超大規模向けの設計を先走りしない。必要になってから対応する
