# 结构图（Structure Map）

> **把一份结构复杂的内容（方法论、教程、SKILL 文档、长文骨架、产品规范）完整可视化为一张知识结构图**——基于组件库自由组装，覆盖原文所有重要节点。
>
> 不适用场景：
> - 文章 / 笔记的核心点凝练 → 用 `visual-card` 插件
> - 活动 / 产品 / 课程的营销宣传 → 用 `poster` 插件

## 命令

```
ikiw map "内容或文件路径" --design <style>                        生成结构图（默认 overview 深度）
ikiw map "内容或文件路径" --depth overview --design <style>        覆盖主要节点，每节点 3-5 项摘要
ikiw map "内容或文件路径" --depth full --design <style>            完整覆盖，所有子项+完整描述（基本是文章重排）
ikiw map "内容或文件路径" --design all                            用所有可用样式各生成一版
```

## 触发词

绝大多数用户不会说"结构图"。agent 应识别以下口语化表达并触发：

- **结构图类**："做个结构图"、"生成结构图"、"完整结构图"、"知识结构图"
- **知识图类**："做个知识图"、"全景知识图"、"知识图谱"、"画个图谱"
- **可视化类**："把这份方法论可视化"、"把这份 SKILL 完整可视化"、"做成一张全景图"
- **完整覆盖类**："涵盖所有内容的图"、"覆盖完整骨架的图"、"把整份文档做成图"、"把整本书做成图"
- **深度类**："深度图"、"完整版图"、"详细结构图"

英文触发词："structure map"、"knowledge map"、"full visualization"、"comprehensive overview"

> 默认 depth = `overview`（覆盖主要节点，更易消化）。用户明确说"完整""详细""所有内容"则切 `full`。

### 不属于本插件的触发场景（要分流出去）

| 用户说 | 应该走哪个插件 |
|--------|---------------|
| "把这篇文章做成图"、"做张要点卡"、"金句卡"、"分享图"、"把这段话做成图发朋友圈" | `visual-card`（内容凝练） |
| "做个海报"、"训练营宣传图"、"产品发布图"、"带二维码的图" | `poster`（营销海报） |

### Agent 主动判断

- 如果用户说"做个图"但内容只有 5-8 个核心点 → 用 `visual-card` 即可，**不要硬塞 structure-map**
- 如果用户说"做个图"且内容是完整方法论 / SKILL / 教程 → 用 `structure-map`
- 不确定时主动询问："这是想浓缩成几个核心点的卡片，还是覆盖所有内容的完整结构图？"

## 何时用哪个 depth

| 场景 | 推荐 depth |
|------|-----------|
| 第一次接触一份方法论，想看全局结构 | `overview` |
| 给团队培训用、需要每个细节都在 | `full` |
| 把一份 SKILL 文档转成可分享的视觉版 | `full` |
| 默认（用户没指定） | `overview` |

## 依赖

- `designs/` — 视觉样式库（design-md 文件）
- `templates/structure-map-portrait.md` — 工作流 + 画布契约 + 组件映射 + 完整性自查
- `templates/structure-map-components.md` — 10 个组件的 HTML 骨架 + CSS 类名 + 字段约束
- `screenshot-export` 插件 — 用于导出为 PNG / PDF

## 执行流程

1. **判断内容类型**：确认原文是结构复杂内容（方法论 / 教程 / SKILL / 长文骨架）。如果是短文/营销 brief，提示用户改用 `visual-card` 或 `poster`
2. **加载规范**：读取 `templates/structure-map-portrait.md` 和 `templates/structure-map-components.md`
3. **解析原文骨架**（必做的预处理）：列出所有 H1/H2/H3，统计每个 H2 下的内容形态（list 条数 / 段落 / 表格）
4. **组件映射**：按 `portrait.md` 的"原文模式 → 组件"表，为每个 H2 选择唯一对应的组件，输出"组件清单"
5. **内容提炼**：按 `--depth` 决定深度
6. **HTML 组装**：按组件清单顺序，从组件库取 HTML 骨架填入
7. **套用视觉样式**：按 `design-md` 的色彩/字体/间距生成 CSS，跨组件统一
8. **完整性自查**：核验原文每个 H2 是否都有对应组件，无遗漏
9. **导出图片**：调用 screenshot-export 插件导出为 3x 高清 PNG

## 关键约定

- **必须先做骨架解析**，再做生成。骨架解析的目的是杜绝"漏内容"
- **组件映射是强契约**：原文模式与组件的对应关系不允许 LLM 凭感觉选
- **完整性 > 美观性**：宁可图很长，不可漏内容（特别是 full 模式）
- **跨组件视觉统一**：所有 section-header / overline 字号字重颜色一致；品牌色只用在固定位置
- 画布契约**优先于** design-md 中的网页 padding 规范
- 自查清单是强制流程，**完整性自查**是 structure-map 特有且最关键的检查项
